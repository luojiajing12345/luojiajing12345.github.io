Cookie和注册登录

发表于 2018-08-30 | 分类于 网络协议

前端可以直接判断输入的账号密码是否符合格式，是否为空，这样就可以不用再发送请求给后端了。
前端也可以不加判断，但是后端一定要判断，因为不一定所有的请求都通过浏览器来发送，如果不是通过浏览器发送的话，前端判断就无效了

Cookie是什么？(重点)

  1. 服务器通过 Set-Cookie 头给客户端一串字符串（登陆不登录都有）
  2. 客户端每次访问相同域名的网页时，必须带上这个字符串
  3. 客户端要在一段时间内保存这个Cookie
  4. Cookie默认在用户关闭页面后就失效，后台代码可以任意设置Cookie的过期时间。

Cookie特点

  1. 服务器通过 Set-Cookie 响应头设置 Cookie
  2. 浏览器得到Cookie之后，每次请求都会带上Cookie(访问同域名)
  3. 服务器读取 Cookie 就知道登陆用户的信息

问题

  1. 我在 Chrome 登陆了，用Safari访问，Safari会带上Cookie吗？
    • 不会，
  2. Cookie存在哪？
    • Window 存在C盘的一个文件里。
    • 其他系统存在硬盘的一个文件中。
  3. Cookie能造假吗？
    • 可以，Chrome的开发者工具——Application——Cookies里面直接改。所以不能相信这个Cookie。
  4. Cookie有有效期吗？
    • 有，但所以时间不确定，默认有效期为20分钟左右，由浏览器决定。例如：浏览器一直没关，Cookie可能一直在。浏览器关了，立马打开，Cookie也还在。浏览器关了，明天再打开，可能Cookie就没有了。
    • 后端可以强制设置有效期，例如一些网站上的，7天内不再重新登录等，可以直接在MDN搜索“Set-Cookie”
      查看文档。

Cookie 如何设置过期时间？如何删除 Cookie？

设置过期时间可以用Expires或Max-Age

  1. Set-Cookie: Expires: Tue, 04 Sep 2018 13:10:29 GMT,2018年9月04号，13：10：29，格林威治时间过期
  2. Set-Cookie: max-age=30s,30秒后过期
    删除Cookie
  3. 使用setcookie()函数删除小部分cookie，
  4. 在浏览器设置里删除所有cookie

前端永远都不要去写或者读Cookie，这是后端做的。

注册到登陆完成流程

  1. 打开注册页面注册
  2. 发送POST请求，带有账号密码
  3. 服务器验证没有问题就会写进数据库
  4. 告诉用户注册成功，用户打开登陆页面
  5. 登陆的时候会发送POST请求
  6. 服务器查询账号密码在不在数据库
  7. 如果在，服务器会给浏览器一个Cookie
  8. 浏览器会把Cookie记录下来
  9. 用户访问该网站的时候，会带上Cookie
  10. 服务器读一下Cookie，查看到里面id，会通过id找到用户信息，会查看到用户的所有信息
  11. 在html页面上填入对应信息，username和password那些，所以就不用再次登录就可以打开网页。
  12. 假设有另一个用户访问该网站，因为没有cookie，所以会给一个所有人一样的页面，让你注册或登陆。

Session

Cookie的问题

假设我们知道数据库里另一个用户的账号，和cookie的组成规则，那么我们只要把我们的cookie修改成另一个用户的cookie，就可以获取到那个用户的密码。
上面说到Cookie是可以改的，那么改了Cookie就变成另一个用户了,这是很不安全的，怎么解决这个问题呢？
答案是：使用Session

什么是Session?

  1. 服务器通过Cookie给用户一个SessionId。
  2. SessionId对应服务器里面的一小块内存，每次用户访问服务器的时候，服务器通过SessionId去读取对应的Session，就知道用户的隐私信息了。
  3. 因为一般SessionId是随机的，所以正常情况下是猜不到的，这样就无法获取用户的Cookie了，也无法改用户的Cookie了。安全由这个随机数保证，没有人能猜到随机数，得不到随机数就得不到Cookie

什么是Session? （重点）

  1. 将SessionID（随机数）通过Cookie发给客户端
  2. 客户端访问服务器时，服务器读取SessionID
  3. 服务器有一块内存（对象）保存了所有的session
  4. 通过SessionID我们可以得到对应用户的隐私信息，如 id、email
  5. 这块内存（哈希表）就是服务器上的所有session
  6. 对服务器来说有很多session,对用户来说有一个session

Session和Cookie的关系

  1. 一般来说，Session是基于Cookie来实现的，因为Session必须把SessionID放到Cookie里面来发给客户端，没有这个SessionID就没有这个Session。

Cookie 和 Session 的区别

  1. Cookie数据放在客户端的系统文件里，每次都随请求发送给 Server
  2. Session 保存在 Server 的内存里，其 Session ID 是通过 Cookie 发送给客户端的
  3. Cookie不安全，因为它数据在本地，容易被别人篡改，session较安全，因为它的数据在服务端，然后一般给的SessionID是随机的，也不容易测试出来。
  4. 一般来说，Session是基于Cookie来实现的，没有Cookie就没有session

localStorage

localStorage简介

  1. localStorage是Html5提供的一个API,里面常用的方法有clear,getItem,setItem,removeItem。
  2. Session是服务器上的哈希表，localStorage是浏览器上的哈希表。
  3. localStorage只能存字符串，直接存函数或对象的话会变成字符串，所有要存对象的话要使用JSON。

localStorage用法

  1. 正常来说js代码中的变量在页面刷新的时候，会全部销毁，刷新后重新声明，没有办法保存下来。
  2. 使用localStorage可以把变量保存下来，刷新页面后也不会销毁，因为localStorage是保存在系统文件里的，这叫持久化存储
  3. 假设我的网站更新了，我要提示用户，但是直接使用aleart()的话，每次刷新这个页面都会弹出这个提示，那我们就可以使用localStorage，先查看localStorage里有没有我们的哈希，如果没有，就进行提示，并把哈希存进去，如果有，就不提示。

localStorage特点（重点）

  1. LocalStorage跟HTTP无关
  2. HTTP 不会带上 LocalStorage 的值
  3. 只有相同域名的页面才能互相读取 LocalStorage（没有同源那么严格）（浏览器的基本功能，这样才安全）
  4. 每个域名LocalStorage最大存储量为5Mb左右（每个浏览器不一样）
  5. 常用场景：记录有没有提示过用户（没有用的信息，不能记录密码，敏感信息）
  6. LocalStorage 永久有效，除非用户清理浏览器缓存数据。

Cookie和localStorage的区别

  1. 每次发送HTTP请求的时候都会带上Cookie给服务器，而LocalStorage不会随http发送给服务器，因为它和http无关。
  2. Cookie的最大存储量一般为4k，而LocalStorage一般为5MB。

SessionStorage（会话存储，和上面的Session没关系）

SessionStorage特点

  1. 1、2、3、4同上，使用方法也一样
  2. SessionStorage 在用户关闭浏览器页面后就失效。

LocalStorage和SessionStorage什么区别？

  1. localStorage在关闭页面甚至浏览器后不会失效（不会自动过期），SessionStorage 在用户关闭浏览器页面(Session结束，网络连接中的会话结束)后就失效（会自动过期）。

HTTP缓存

Cache-Control

  1. 服务器某个响应部分加上’Cache-Control’,’max-age=30’（单位是秒）
  2. 第一次请求还是会下载，如果客户端再次请求这个相同的url，浏览器就会阻断这个请求，直接从浏览器的内存里或者硬盘里返回上一次请求到的文件。
  3. 过了30s后再次发起请求才不会被阻断。（在测试的时候不要勾选浏览器上的Disable cache）
  4. 不要给首页（html）设置缓存，因为如果首页都设置了缓存，那么浏览器不会给服务端发送任何请求，如果代码更新了，无法获取最新的网页，所以我们要留一个入口。

如何更新缓存

  1. 假设我们设置的缓存时间比较长，但是网页的js更新了，怎么办？
  2. 稍微改动一下请求的url，例如./css改成./css?v=2,或者改成./css?随机数。

Expires

  1. 从前的更新缓存是使用Expires：response.setHeader('Expirse', 'SUM,04 Feb 2018 14:04:04 GM'),2018年二月24好，14点4分5秒，格林威治时间过期，过了以后就没缓存了。但是这是根据自己客户端的本地时间的，如果时间错了，就会出问题。
  2. 如果同时有Cache-Control和Expirse，会优先使用Cache-Control

Etag

MD5

  1. MD5-摘要算法，就是把一个文件变成一个字符串，如果文件里的内容有改一点点，算出来的md5，差异都很大。差异越小，算出来的md5差别越大。
  2. 例如下载个Linux.iso 400M,怎么知道下载的完不完整呢，万一少一两个字节怎么办？
  3. 网上的Linux.iso可以算个MD5，然后我本地下载后也算一个MD5，进行对比，一样就代表完整。

Etag的使用

  1. 安装个MD5npm install MD5，引入nodejs里面
  2. 给访问的文件的响应头部加上Etaglet string = fs.readFileSync('./js/main.js', 'utf8'), response.setHeader('ETag', fileMd5)。
  3. 在第一次请求的时候请求响应头会有个Etag:8ae0c72f0ceab4c0511cd69747342af3，这个值就是它的md5,刷新一次，第二次请求的请求头上会有个（如果不匹配）If-None-Match: 8ae0c72f0ceab4c0511cd69747342af3。
  4. 借助请求头，我们直接用if判断我们需要的文件的请求头上的md5和服务器上的请求文件的md是否相等，如果相等，则http状态码为304，代表从上次请求后，请求的网页为修改，服务器返回响应后，不会返回网页内容。

Cache-Control和Etag

  1. Cache-Control直接不请求，直接读取内存或磁盘里的数据
  2. Etag请求但不返回内容，查看md5是否相等，相等就不返回内容
  3. Cache-Control比较好，连http请求都不用发送了。

Cache-Control: max-age=1000 缓存 与 ETag 的「缓存」有什么区别？

  1. Cache-Control:max-age=1000的话，第一次请求过后，会把响应内容进行缓存，缓存内容会在1000秒后失效，1000s内不会再访问相同的url，直接从内存或磁盘里读取缓存内容。
  2. Etag是在文件HTTP响应头上添加的标识符。客户端请求后，在请求相同url的http请求头中，会有个If-None-Match的属性，它的值对应请求文件的值，如果相等，则服务器直接返回http状态码304，代表内容没有改变，不会返回请求内容，直接使用缓存的内容。只有改变了请求的文件，Etag的值和If-None-Match的值不相等，服务器才会返回http状态码200，并返回相应内容。
  3. 最大的区别是Cache-Control:max-age=1000，在1000s内不会发送http请求包，而Etag每次请求都会发送http请求包。
