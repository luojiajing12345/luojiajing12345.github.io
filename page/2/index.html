##高度是由什么决定的？

内联元素：主要是由建议行高决定，我们只能决定字体的高度，无法决定包裹字体元素的高度，这些主要由设计这些字体的设计师决定。每种不同的字体，会使包裹它的元素的高度不同
字体三部分：基线，上部，下部，还有个建议行高。每种字体的建议行高是不一样的，文字都会基线对齐。
把一个文字设置为100px，例如“hug”，从h的最上边到g的最下边为100px。
例如把span里的文字设为100px，那么span的高度会比100px高，因为会有个建议行高，每种字体的默认行高不一样。
line-height能使文字自动垂直居中。

块级元素:高度有其内部文档流元素的高度总和决定。决定，但不相等。（假设body里什么都没有，那么他的高度是0（只有个回车不算））

##文档流是什么？
文档内元素的流动方向。
内联元素从左往右流，如果流动遇到阻碍，那就换行继续从左往右流动。
如果一个元素例如，中间类截断，假设这个元素有加边框，那么那个元素会被分开，但还是一个元素，边框也还是一个，只是头在第一行，身体在第二行。
如果被截断的是英文单词，那么英文单词并不会换行，会缩到里面去。因为中文每个字是一个东西，而英文单词也是一个东西，所以默认不会被分开。如果要想使其可以被截断：

添加css属性:word-break:break-all。

块级元素从上往下流，每一块都占一行，新的一块换新一行。
把块级元素也变成从左往右流：display：inline-block（不推荐使用，建议用floot浮动）

position fixed

position属性用于指定一个元素在文档中的定位方式。
相对定位：relative。
绝对定位：absolute（基于祖先元素定位）和fixed(基于屏幕视口定位，元素位置在屏幕滚动时不会改变，会创建新的层叠上下文，脱离文档流)。
以下是基于屏幕左上角视口定位，创建新的层叠上下文

div {
position fixed;
top 0;
left 0;
}

脱离文档流，使用position fixed后，该元素会脱离文档流，脱离文档流后，其父级元素的高度就不受其影响了，因为块元素的高度为其内部文档流高度总和，脱离后就没关系了
当使用position fixd后，此元素就不会往外扩展了，它有多大，就占用多大范围。

HTTP状态码详解

Posted on 2018-04-25 |

1xx消息

这一类型的状态码，代表请求已被接受，需要继续处理。这类响应式临时响应，只包含状态航和某些可选的响应头信息，并以空行结束。由于HTTP/1.0协议中没有定义任何1xx状态码，所以除非在某些试验条件下，服务器禁止向此类客户端发送1xx响应。这些状态码代表的响应都是信息性的，标示客户应该采取的其他行动。

2xx成功

这一类型的状态码，代表请求已成功被服务器接收、理解、并接受。

3xx重定向

这类状态码代表需要客户端采取进一步的操作才能完成请求。通常，这类状态码用来重定向，后续的请求地址（重定向目标）在本次响应的Location域中指明。

4xx客户端错误

这类的状态码代表了客户端看起来可能发生了错误，妨碍了服务器的处理。除非响应的是一个HEAD请求，否则服务器就应该返回一个解释当前错误状况的实体，以及这是临时的还是永久性的状况。这些状态码适用于任何请求方法。浏览器应当向用户显示任何包含在此类错误响应中的实体内容。
如果错误发生时客户端正在传送数据，那么使用TCP的服务器实现应当仔细确保在关闭客户端与服务器之间的连接之前，客户端已经收到了包含错误信息的数据包。如果客户端在收到错误信息后继续向服务器发送数据，服务器的TCP栈将向客户端发送一个重置数据包，以清除该客户端所有还未识别的输入缓冲，以免这些数据被服务器上的应用程序读取并干扰后者。

5xx服务器错误

表示服务器无法完成明显有效的请求。这类状态码代表了服务器在处理请求的过程中有错误或者异常状态发生，也有可能是服务器意识到以当前的软硬件资源无法完成对请求的处理。除非这是一个HEAD请求，否则服务器应当包含一个解释当前错误状态以及这个状况是临时的还是永久的解释信息实体。浏览器应当向用户展示任何在当前响应中被包含的实体。这些状态码适用于任何响应方法。

HTTP入门

Posted on 2018-04-25 |

历史

1990年，Tim Berners-Lee（李爵士）发明了用网址就能访问网页的办法，他发明了第一个网页、第一个浏览器和第一个服务器。
WWW（World Wide Web）万维网，也是李爵士发明的。
主要包含三个概念：

  1. URI，俗称网址。
  2. HTTP，文本传输协议（主要用来传输内容）。
  3. HTML，超文本标记语言。

URI的作用是能让你访问一个页面，HTTP的作用是让你能下载这个页面，HTML的作用是让你能看懂这个页面。

URI分为URL（统一资源定位符）和URN（统一资源名称），我们一般使用URL作为网址，URN的作用是通过提供一种途径，用于在特定的名字空间资源的标识，以补充网址。

URL的常见组成：

https://www.baidu.com/s?wd=hello&rsv_spt=1#5
协议/域名/路径/查询参数/锚点

DNS（域名系统）

dns作为将域名与ip地址相互映射的一个分布式数据库，能够使人更方便地访问互联网。使用53端口。

HTTP的请求与响应。

  1. 客户端负责发起请求
  2. 服务端在80端口接收请求
  3. 服务端负责返回内容（响应）
  4. 浏览器负责下载响应内容
    HTTP的作用就是知道浏览器和服务器如何沟通。

##请求的格式
看起来那么多，，格式都差不多的。

1 动词 路径 协议/版本
2 Key1: value1
2 Key2: value2
2 Key3: value3
2 Content-Type: application/x-www-form-urlencoded
2 Host: www.baidu.com
2 User-Agent: curl/7.54.0
3
4 要上传的数据

  1. 请求最多包含四部分，最少包含三部分。（也就是说第四部分可以为空）
  2. 第三部分永远是一个回车（\n）
  3. 动词有GET POST PUT PATCH DELETE HEAD OPTIONS等。
  4. 这里的路径包括[查询参数]，但不包括[锚点]。
  5. 如果你没有写路径，那么路径默认为/。
  6. 第二部分中的Content-Type标注了第四部分的格式。
  7. 如果有请求的第四部分，那么在FormData或Payload里面可以看到。

响应

请求之后，除非断网或者服务器宕机了，否则都应该能的要一个响应。
响应的格式：

1 协议/版本号 状态码 状态解释
2 Key1: value1
2 Key2: value2
2 Content-Length: 17931
2 Content-Type: text/html
3
4 要下载的内容

用Chrome查看响应

  1. 打开 Network
  2. 输入网址
  3. 选中第一个响应
  4. 查看 Response Headers，点击「view source」，点击「view source」，点击「view source」
  5. 你会看到响应的前两部分
  6. 查看 Response 或者 Preview，你会看到响应的第 4 部分

HTTP状态码，状态码是服务器对浏览器说的话。

  • 1xx消息——请求已被服务器接收，继续处理。
  • 2xx成功——请求已成功被服务器接收、理解、并接收。
  • 3xx重定向——需要后续操作才能完成这一请求。
  • 4xx请求错误——请求含有词法错误或者无法被执行。
  • 5xx服务器错误——服务器在出路某个正确请求时发生错误

脚本

什么是脚本？

脚本原来自于戏剧舞台，脚本主要由任务对话和舞台提示组成，演员和道具组只需按照脚本说的做即可。
编程领域的脚本程序也是如此，主要由脚本语言和注释组成，计算机只要按照脚本上的做即可。例如：

cd ~/Desktop
mkdir demo
cd demo
echo “hi” > index.html
cd ~/Desktop

各个领域的脚本有各个领域的脚本的语法，但功能都差不多，只是语法不同。

Hexo+GitHub搭建博客和修改主题和绑定域名

Posted on 2018-04-24 |

搭建博客

  1. 使用Git Bash进入一个安全的目录。

  2. 在GitHub上新建一个空repo，repo名称是 [你的github用户名.github.io]。

  3. npm install -g hexo-cli，安装Hexo。

  1. Hexo init myBlog,新建个目录并初始化

  2. cd myBlog。

  3. npm i，安装所有模块。

  4. hexo new 博客章节名，会看到一个md文件的路径。

  5. start xxxx.md,编辑这个md文件，windows下直接进目录打开操作也行。

  6. 打开_config.yml，这个是站点配置文件，这里可以修改网站配置。

    把第6行的title改成想要的名字。
    把第9行的author改成自己的名字。
    把最后以行的type改成type: git。
    在最后一行后面新增一行，左边与type平齐，加上一行repo: 仓库地址，（请将仓库地址改为「你的用户名.github.io」对应的仓库地址，仓库地址以 git@github.com: 开头的）。
    repo和type后面都有个空格。

  7. npm install hexo-deployer-git --save，安装git部署插件

  8. hexo deploy,

  9. 进入[你的用户名.github.io]对应的repo，打开GitHub Pages功能，如果打开了，就直接点击预览链接就能看到博客了。

以后上传博客的方法

  1. 打开Git Bash终端，进入存放hexo的目录。
  2. hexo new 博客名，新建个md文件。
  3. 编辑上面新建的文件。

  4. hexo generate==hexo g，生成

  5. hexo deploy==hexo d，部署

修改主题

  1. 主题大全
  2. 知乎主题
  3. 找到想要的主题后进入主题的GitHub首页，例如最火的https://github.com/iissnan/hexo-theme-next
  4. 复制它的SSH地址或HTTPS地址，假设地址是git@github.com:iissnan/hexo-theme-next.git

  5. cd thems进入thems目录。

  6. git clone git@github.com:iissnan/hexo-theme-next.git，把所有代码下载到当前目录。

  7. cd ..，返回上层

  8. 将 _config.yml 的第 75 行改为 theme: hexo-theme-next(这个是下载下来的主目录的名字)，保存。

  9. hexo g，

  10. hexo d，

  11. 等一分钟，刷新博客，就能看到新的主题了。

  12. 创建一个github新仓库，用来保存myBlog里面的“生成博客的程序代码”
  13. 连接好新仓库后，每次hexo d生成新博客后，都add/commit/push一下[生成博客的程序代码]，传到github上去。
  14. 一共有两个仓库，一个是保存博客的，一个是保存生成博客的程序代码的。

绑定域名

  1. 前往卖域名的网站购买域名（国内的要备案，国外的xxx）
  2. 使用电脑ping一下自己的博客的github链接，就是那个xxxx.github.io的，千万不要加https这些前缀，ping了以后会显示个ip地址，记住那个IP地址。
  3. 前往阿里云（腾讯云什么的也行），登陆进去后首先找到控制台–>产品与服务–>域名–>域名列表，设置域名解析，首次可以点新手引导设置。把刚才记下的ip与这个域名绑定就好了。
  4. 在博客的github链接，就是xxx.github.io的根目录哪里添加个CNAME，里面填写自己的域名。
  5. 在Hexo目录的source目录下新建个CNAME的文件，注意这个文件是没有后缀的，里面也填写自己的域名。例如：

    luojj.com

  6. 添加完后在浏览器输入自己的域名，打开的就是github上的博客了。

注册与登陆

  1. 打开网址“www.github.com”
  2. 注册账号，登陆账号

配置github

  1. 前往https://github.com/settings/keys上添加新key。
  2. title随便写，由于我们没有key，所以要新建个key。
  3. 打开终端（Git Bash）输入ssh-keygen -t rsa -b 4096 -C "你的邮箱"(生成一堆文件在/.ssh目录下，id_rsa相当于钥匙，id_rsa.pub相当于锁)
  4. 运行 cat ~/.ssh/id_rsa.pub，得到一串东西，复制粘贴到key里，点击 Add SSH key
  5. 运行 ssh -T git@github.com

配置git

依次运行
git config --global user.name “英文名”
git config --global user.email “邮箱”
git config --global push.default matching或(simple)（matching匹配所有分支）（simple匹配单个分支）
git config --global core.quotepath false
git config --global core.editor “vim”
global前面两个-

使用git的三种方法

  1. 只在本地使用
    初始化本地仓库git init，初始化后里面有个.git的文件夹

将变动文件添加到暂存区git add 文件路径

正式提交变动文件，提交到.git仓库git commit -m "注释信息"

如果有新的变动我们只需依次执行git add xxx和git commit -m "xxx"两个命令即可。

记住：先add再commit即可

查看变更历史git log

  1. 将本地仓库上床单GitHub
    在GitHub上新建一个空仓库（除了仓库名，其他都别改），然后会进入一个很多命令的页面
    点击ssh按钮获取ssh地址。
    然后复制代码再运行：git remote add origin git@github.com:xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx/xxx.git

复制第二行代码再运行：git push -u origin master
刷新当前页面，仓库就上传到GitHub了。

  1. 下载GitHub上的仓库
    在GitHub上新建一个仓库（不是空仓库，要勾选Initialize那个按钮），仓库自带了README。
    仓库有了几个文件了。
    点击绿色按钮[clone or download],会看到一个弹出层。
    确保弹出层里的地址是SSH地址，点击Use SSH按钮，然后复制这个地址。
    打开Git Bash，进入一个目录后，运行git clone 你刚才得到的以git@github.com开头的地址,运行完会发现当前目录下多了个目录。

如果在本地目录有任何变动，只需按照以下顺序就能上传：

  1. git add 文件路径
  2. git commit -m “信息”
  3. git pull (增量下载，如果远程仓库有变化要使用这个把仓库的下载下来以后才能上传上去)
  4. git push（上传）

HTML5的内容类型

内嵌：在文档中添加其他类型的内容，如audio、video、canvas和iframe等。
流：在文档中和应用的body中使用的元素，如form、h1和small等
标题：段落标题，如h1、h2和group等
交互：与用户交互的内容，如音频和视频的控件、button和textarea等
元数据：通常出现在页面的head中，设置页面其他部分的表现和行为，如script、style和title等。
短语：文本和文本标记元素，如mark、kbd、sub和sup等。

HTML5的基础元素

<!DOCTYPE>:定义文档类型。这个标签可告知浏览器文档使用哪种html或xhtml规范。
作用 ：声明文档的解析类型（document.compatmode），避免浏览器的怪异模式。
document.compatMode:
BackCompat:怪异模式，浏览器使用自己的怪异模式解析渲染页面。
CSS1Compat：标准模式，浏览器使用W3C的标准解析渲染页面。

声明的重要性：

这个属性会被浏览器识别并使用，但是如果页面中没用DOCTYPE的声明，那么compatMode默认就是BackCompat(怪异模式)。浏览器就会按照自己的方式解析渲染页面，那么，在不同的浏览器就会显示不同的样式
如果页面添加了<!DOCTYPE html>，那么，浏览器就会按照W3C的标准解析显然页面，这样，页面所在的浏览器里显示的就都是一个样了。

<html>:定义html文档

<head>:定义文档的头部，包括了当前文档的元数据,元数据就是描述数据的数据，比如，文档的标题、使用的字符集、关键字等信息，这些信息并不会真正被浏览器所渲染。

<body>:定义文档的主体，heml文档的内容都在这里边，一个文档只允许有一个<body>标签。

:定义文档的标题，即浏览器窗口显示出的标题。

<h1>to<h6>：定义html文本标题，依次从大到小。

<p>:定义段落。
<br>: 换行。

<hr>: 定义水平线。
<!–…-->:定义注释。
